f:T1fc6,<p>크로스 브라우징은 HTML, CSS, Javascript 작성 시 W3C의 웹 표준 규격에 맞도록 작성하며, 제작한 웹페이지가 서로 다른 환경을 가진 브라우저들에서 의도한 대로 작동하도록 하는 작업을 말한다.</p>
<p>브라우저 마다 <a href="https://github.com/da-in/tech-interview-study/blob/main/Web/%EB%B8%8C%EB%9D%BC%EC%9A%B0%EC%A0%80%EC%99%80%20%EB%A0%8C%EB%8D%94%EB%A7%81.md">랜더링 엔진</a>이 다르기 때문에, 작동되지 않는 HTML5, JS 코드와 해석하지 못하는 CSS 코드가 존재할 수 있다. 브라우저에 따른 버그가 존재할 수 있고, 브라우저 자체 CSS 차이의 영향을 받을 수 있다.</p>
<p>국내 블로그 자료들에서는 <strong>크로스-브라우징</strong>(Cross Browsing)이라는 표현을 많이 사용하였으나, 해외에서는 아래의 <strong>크로스-브라우저 테스팅</strong>(Cross-browser Testing), <strong>크로스-브라우저 호환성</strong>(Cross-browser Compatibility)의 표현을 주로 사용하였다.</p>
<p><strong>Cross-browser Compatibility</strong><br>
Cross Browsing을 통해 보장하고자 하는 브라우저 간의 호환성을 의미한다.</p>
<p><strong>Cross-browser Testing</strong><br>
Cross-browser Compatibility를 보장하기 위해 진행하는 <strong>테스팅 기법</strong>을 뜻하며 일반적으로 <strong>QA 엔지니어</strong>에 의해 수행된다.<br>
글꼴 및 이미지의 표시, 반응형 웹 디자인 작동 여부 등 <strong>콘텐츠 및 레이아웃의 일관성</strong>을 테스트한다.<br>
기능, third-party services와의 통합, 모바일 또는 태블릿용 터치 입력과 같은 <strong>유용성</strong>을 확인한다.<br>
이미지의 대체 텍스트, 비디오의 폐쇄 자막(표시 여부를 선택할 수 있는 자막) 등의 <strong>접근성</strong>을 테스트 한다.<br>
<a href="https://developer.mozilla.org/en-US/docs/Learn/Tools_and_testing/Cross_browser_testing#prerequisites">MDN Web Docs - Cross browser testing</a></p>
<h2>크로스 브라우징에 대한 오해</h2>
<p>크로스브라우징은 완전한 동일성을 의미하는 것이 아니라 최소한 핵심 기능의 사용은 보장하는 동등성을 의미한다. 페이지가 완전히 똑같이 보이게 하는 것이 아님에 유의해야한다.<br>
예를 들어 동일한 페이지가 특정 브라우저에서 더 좋아 보일 수 있다. 하지만 정도의 차이가 있더라도 모든 브라우저에서 읽는 행위는 가능해야한다.</p>
<blockquote>
<p>ex - Modern Browser에서의 입체 애니메이션과, 구형 브라우저에서의 평면 그래픽.</p>
</blockquote>
<h2>Cross Browsing 을 위한 방법</h2>
<ol>
<li>브라우저와 렌더링엔진의 점유율을 파악하고, 점유율이 높은 브라우저와 렌더링 엔진을 타겟팅하여 우선적으로 대응한다.</li>
<li>사용하는 속성의 브라우저간 호환성을 확인한다.<br>
<a href="https://caniuse.com/">https://caniuse.com/</a> 등의 사이트에서 확인 가능하다.</li>
<li>Cross Browser Testing을 진행한다.<br>
수동으로도 가능하지만, LambdaTest, CrossBrowserTesting.com 등의 온라인 사이트를 이용하는것이 편리하다.</li>
<li><strong>Reset.css</strong>를 이용하여 CSS 초기화 작업을 수행한다.<br>
<code>Reset.css</code>를 검색하면 많은 자료들이 나온다. Reset.css는 모든 속성에 대한 default 값을 한번씩 지정해서 브라우저마다 다른 기본 스타일 값을 초기화해주는 것이다. 그러면 모든 브라우저에서 동일한 스타일 결과를 기대할 수 있다.</li>
<li>Vender Prefix<br>
CSS 속성 앞에 브라우저 별 접두사(Vender Prefix)를 붙이는 것이다. Vender Prefix는 과거 특정 브라우저에서만 지원되는 속성을 사용하기 위해 제공된 기능이다. 브라우저는 자신의 Vender Prefix가 붙은 속성을 인식하고 적용한다.<br>
<a href="https://autoprefixer.github.io/">autoprefixer.github.io</a>는 접두사를 붙인 CSS를 생성해준다.</li>
</ol>
<h4>Vender Prefix와 사용 예</h4>
<p>| prefix   | browser                                                                                                                                      |

| -moz-    | Firefox                                                                                                                                      |
| -ms-     | IE                                                                                                                                           |
| -o-      | Opera의 WebKit 이전 버전                                                                                                                     |</p>

<h2>렌더링 엔진(레이아웃 엔진)</h2>
<p>서로 다른 브라우저여도 동일한 렌더링 엔진을 사용한다면 비슷한 결과를 얻을 수 있다.</p>
<p><img src="https://user-images.githubusercontent.com/66757141/206547114-5d26cf01-dfc0-4867-8641-3547e9e38570.png" alt="화면 캡처 2022-12-09 041556">
<em>Comparison of browser engines <a href="https://en.wikipedia.org/wiki/Comparison_of_browser_engines">Wikipedia</a></em></p>
<p><strong>Active Status</strong>는 새로운 웹 표준이 엔진에 추가되어 멀티미디어를 포함한 대다수의 웹사이트를 렌더링할 수 있음을 의미한다.<br>
<strong>Maintained Status</strong>는 적어도 엔진이 여전히 컴파일 됨을 의미한다.<br>
<strong>Discontinued</strong>는 엔진의 제공이 중단되었음을 의미한다.</p>
<ul>
<li><strong>Webkit</strong> | 초기 애플사가 사파리 엔진으로 사용하기 위해 차용했으나 현재는 웹킷 프로젝트로 분리되어 개발되고 있다.<br>
크롬에서도 사용되었던 엔진이며 iOS나 안드로이드의 기본 브라우저들이 이 웹킷 엔진을 사용한다.<br>
점유율이 높은 엔진이다.</li>
<li><strong>오페라</strong>는 15버전부터 프레스토가 아닌 블링크를 사용했다.</li>
<li><strong>듀얼 엔진</strong> 국내 이스트소프트의 스윙(Swing) 브라우저를 예로 들면 크롬과 같은면서도 액티브X를 지원하는데, 독자적인 엔진이 아닌 두가지 엔진을 번갈아 사용하는 방식이기 때문에 가능하다. - 보통 트라이던트와 웹킷or블링크 엔진을 사용한다.</li>
</ul>
